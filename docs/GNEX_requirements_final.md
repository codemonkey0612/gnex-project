## 1. プロジェクト概要

### 1.1 プロジェクトの目的
建物オーナー・管理者（発注者）と、エネルギー・産廃関連の専門業者（受注者）をつなぐBtoBマッチングプラットフォームの構築。

「環境ビジネス」という視点ではなく、建物のコスト削減、およびCO₂排出削減にフォーカスしたワンストッププラットフォームを目指す。

> **※田邊様メモ：** コスト削減は手段であって、本当の目的は脱炭素という認識。しかし企業はコスト削減にフォーカスした方がサイトを利用する明確な理由付けができるため、脱炭素は少しのアピール箇所にとどめる。

### 1.2 事業コンセプト
- 建物の課題・悩み → コスト削減効果シミュレーション → 導入比較検討 → 導入費の適正化 → プロジェクト完了
- 「環境」という言葉を極力使わず、ビジネスとしてのコスト削減を全面に訴求
- カラーテーマ：青系・紺系（#1F3864 / #2E75B6）

### 1.3 対象ユーザー（MVP）
- **法人：** 建物を所有・管理する企業・法人
- **個人：** マンションオーナー等の個人（対象）
- **対象外（MVP）：** 戸建ての個人 → MVP後に拡張検討

### 1.4 サービス名・ブランド方針
- サービス名：**G-NEX**（"GX Matching Platform"）
- カラー：青系・紺系
- 「環境」という言葉は使用しない。「コスト削減」「エネルギー最適化」を軸にした言葉選び

## 5. 機能要件

### 5.1 発注者画面

#### 案件投稿フォーム（大幅追記）

**フォームUI：タップ選択式のウィザード形式（ステップ式）**

- Step1：カテゴリー選択（Unit A〜D）
- Step2：建物種別・所在地・予算目安の入力
- Step3：写真・図面アップロード
- Step4：確認・送信

**フォームの詳細仕様：**
- `<input capture>` を使用したカメラ直接起動（モバイル対応）
- **動的フォーム生成（マスタ管理）：** カテゴリーごとにフォーム項目を切り替え
  - 管理画面から「写真項目の増減」「質問文の増減・修正」「選択肢の増減」が可能
  - 完全自由追加は必須としない（テキスト修正・選択肢変更レベルの簡易管理、またはコード保守対応）
- **一時保存機能（Cookie/LocalStorage）：** ブラウザを閉じても入力内容を保持
- **専門家マッチング分岐：** 「工事の依頼」だけでなく「補助金・助成金の相談（専門家への依頼）」の選択肢を設置
- **非公開チェックボックス：**
  - 「□ 案件を非公開にする」
  - 「□ 写真を非公開にする」
  - ※裏側：管理者に通知
- **リード課金チェックボックス（Step1 または マイページ）：**
  - 「□ 補助金・助成金の活用相談を希望する」
  - 「□ 初期費用0円での導入（リース・分割払い・PPA）を希望する」
  - 「□ 廃棄・有価物買取（産廃診断）を希望する」
  - ※チェックが入った案件はDB上で「リード販売対象案件フラグON」として処理

**写真アップロード仕様（田邊様追記）：**
- **写真の後付けアップロード対応**（投稿後も追加可能）
- **自動圧縮・リサイズ：** スマホ撮影画像（3〜5MB）を1/10程度まで縮小・圧縮
- **1投稿あたりの上限：** 写真1枚最大10MB、1回の投稿で合計100MBまで
- **画像保存先：Amazon S3（S3連携）**
  - S3のライフサイクルポリシー：1年経過した古いデータは自動的にGlacier等へ移動
- **写真枚数：** Figmaの4枚から増枚可能（枚数は要確認）
- **写真撮影ガイド機能（CMS連携）：**
  - CMSからカテゴリーごとに「撮影してほしい写真項目」を設定
  - 1写真項目につき：①写真タイトル ②必須/任意チェック ③撮影ガイド画像・テキスト
  - フロント側：選択カテゴリーに紐付けられた写真項目群が動的生成・表示
  - 「💡 撮影のポイント」として良い例・悪い例のガイド画像（JPEG）とテキストを表示
  - ガイド画像：管理者がCanva等で作成した1枚画像をCMSからアップロードするだけのシンプル仕様

**免責事項の表示：**
見積提示画面に以下の免責事項を表示：
> 「※この見積りは、頂いた情報に基づく簡易的な試算です。正式な金額は、必ず現地調査後に確定します。現地の状況により、金額が大きく変わる可能性がありますことを予めご了承ください。」

**AI入力サポート機能：** 依頼内容の記載補助（検討・実装）

## 6. リード課金システム

### 6.1 動線設計（発注者側）
1. 発注者が案件登録（Step1）またはマイページの見積比較画面で「付加価値ニーズ」チェックボックスにチェック
2. チェックが入った案件はDB上で「リード販売対象案件（フラグON）」として処理

**対象リード種別：**
- ① 補助金ニーズ → 対象：士業（行政書士・中小企業診断士等）
- ② 融資・リース・PPAニーズ → 対象：地方銀行・信用金庫・中堅リース会社
- ③ 産廃診断ニーズ → 対象：有価物買取業者

### 6.2 自動通知・マッチング
- リード販売対象案件発生時、管理画面で属性登録されている該当業者アカウント宛に自動通知メール送信
- 通知内容は匿名情報のみ：「大阪府 / 製造業 / 太陽光導入予定 / 概算規模○○万円」
- 発注者の「企業名」「詳細な住所」「連絡先」は完全に伏せる

### 6.3 受注者（リード購入側）の動線
1. 通知受信後、G-NEXにログイン
2. 「リード案件一覧」から該当案件の詳細ページを開く
3. **「この案件の連絡先を開示する（料金：○○円）」ボタン**をクリック
4. **Stripe Checkout（都度決済）** へ遷移 → クレジットカード決済完了後、発注者の企業名・連絡先が表示

### 6.4 決済方式
**採用：案A - Stripe都度決済**
- 未回収リスクなし
- 発注者連絡先開示までの動線がシームレス
- Stripe Checkout使用

### 6.5 管理者向け機能
- リード購入用アカウントの属性管理（士業/金融/産廃）
- カテゴリーごとのリード単価設定（管理者が変更可能な変数）
- 当月のリード購入件数・金額の一覧・集計画面

## 7. 統合型シミュレーター仕様（田邊様追記）

**設計方針：** 計算式はプログラムに固定せず、管理画面（CMS）から6つの変数を設定できる汎用マスタ構造で実装。将来的な商材追加・補助金制度変更にシステム改修なしで対応。

### パターンA：規模・数量ベース
対象：Unit A（太陽光）、Unit C（EV充電）など、面積や台数を入力するもの

**管理画面で設定する6つの変数：**
- 変数1（規模換算係数）：入力値を基準単位（kW等）に変換する数値
- 変数2（投資単価）：基準単位あたりの工事・設備費用
- 変数3（補助率）：補助金が適用される割合（例：1/3なら0.33）
- 変数4（補助金上限額）：補助金の最大支給額
- 変数5（年間削減単価）：基準単位あたりの年間コスト削減額
- 変数6（CO2削減単価）：基準単位あたりの年間CO2削減量

**計算式：**
1. 基準規模 = ユーザー入力値 × 変数1
2. 概算投資額 = 基準規模 × 変数2
3. 補助金概算 = MIN（概算投資額 × 変数3、変数4）
4. 年間削減額 = 基準規模 × 変数5
5. 投資回収年数 = （概算投資額 − 補助金概算）÷ 年間削減額
6. 年間CO2削減量 = 基準規模 × 変数6

### パターンB：現状コストベース
対象：Unit B（高効率空調・LED）、Unit D（産廃・保守）など、金額を入力するもの

**管理画面で設定する6つの変数：**
- 変数1（年間換算係数）：入力金額を「年額」にするための数値（月額入力なら12）
- 変数2（投資額倍率）：現状の年間コストの何倍が初期投資になるかの目安
- 変数3（補助率）：補助金が適用される割合
- 変数4（補助金上限額）：補助金の最大支給額
- 変数5（コスト削減率）：コストが何割下がるか（例：30%削減なら0.3）
- 変数6（CO2換算係数）：削減できた金額1円あたりの概算CO2削減量

**計算式：**
1. 現状の年間コスト = ユーザー入力値 × 変数1
2. 概算投資額 = 現状の年間コスト × 変数2
3. 補助金概算 = MIN（概算投資額 × 変数3、変数4）
4. 年間削減額（または収益）= 現状の年間コスト × 変数5
5. 投資回収年数 = （概算投資額 − 補助金概算）÷ 年間削減額
6. 年間CO2削減量 = 年間削減額 × 変数6

### 設定例（管理画面での入力イメージ）

**設定例1：太陽光発電（パターンA）**
- ユーザー入力：屋根の広さ（平米）
- 変数1：0.15（1平米あたり0.15kW）
- 変数2：250,000（1kWの工事費25万円）
- 変数3：0.33（補助金は1/3）
- 変数4：50,000,000（上限5000万円）
- 変数5：30,000（1kWあたり年間3万円削減）
- 変数6：0.45（1kWあたり年間0.45トンCO2削減）

**設定例2：高効率空調・LED（パターンB）**
- ユーザー入力：現在の月間電気代（円）
- 変数1：12（年額換算）
- 変数2：2.5（年間電気代の約2.5倍が工事費概算）
- 変数3：0.33（省エネ補助金1/3）
- 変数4：20,000,000（上限2000万円）
- 変数5：0.3（電気代30%削減）
- 変数6：0.000017（電気代1円削減あたりのCO2削減効果）

**設定例3：EV充電設備・V2H（パターンA）**
- ユーザー入力：充電器の台数（台）
- 変数1：1
- 変数2：1,500,000（1台あたり150万円）
- 変数3：0.5（50%補助）
- 変数4：5,000,000（上限500万円）
- 変数5：100,000（1台あたり年間10万円燃料費削減）
- 変数6：1.5（1台あたり年間1.5トンCO2削減）

**設定例4：産廃・有価物買取（パターンB・特殊ロジック）**
- ユーザー入力：現在の月間産廃処理費（円）
- 変数1：12
- 変数2：0（初期投資0円）
- 変数3：0（補助金なし）
- 変数4：0（上限なし）
- 変数5：0.5（処理費の50%が有価物利益に変わる）
- 変数6：0.00001（リサイクルによるCO2削減）
- ※変数2を「0」にするだけで「投資0円で利益が出る」特殊計算が自動成立

## 8. LP（ランディングページ）要件

### 8.1 訴求フロー（ユーザージャーニー）
1. 建物の課題・悩み（共感）
2. コスト削減効果シミュレーション（興味喚起）
3. 導入の比較検討（Unit A〜Dの具体的提案）
4. 導入費の適正化（補助金・PPA・リース）
5. プロジェクト完了・問い合わせCTA

### 8.2 建物別カテゴリー（確定版）

| # | カテゴリー名 | ターゲット属性 | 提案商材（フック→本丸） |
|---|---|---|---|
| 1 | 工場・倉庫【最重要】 | 製造業・物流 | Unit D（廃プラ買取）→ Unit A（屋根上太陽光） |
| 2 | 病院・介護施設【最重要】 | 医療・福祉法人 | Unit D（医療ゴミ・オムツ）→ Unit A（BCP蓄電池） |
| 3 | 店舗・商業施設 | 小売・スーパー・学校 | Unit D（食品ゴミ・梱包材）→ Unit A（屋根上太陽光）・Unit B（冷凍機・空調） |
| 4 | ホテル・旅館 | 宿泊業 | Unit B（ボイラー・空調）→ Unit C（EV充電器） |
| 5 | ビル・マンション | 不動産・管理組合 | Unit C（EV充電器）→ Unit B（共用部LED） |
| 6 | その他施設 | 学校・データセンター・ゴルフ場・駐車場 | EV充電器、LED |

**建物別詳細ターゲット：**
- 工場・倉庫：食品工場、金属加工、物流センター
- 病院・介護：総合病院、クリニック、特養ホーム
- 商業施設：スーパーマーケット、ドラッグストア、ショッピングモール、スポーツジム、パチンコ店、カーディーラー
- ホテル・旅館：ビジネスホテル、温泉旅館
- ビル・マンション（統合）：テナントビル、賃貸マンション、分譲管理組合

### 8.3 課題別カテゴリー（Unit別）
- Unit A：創エネ・蓄エネ（太陽光・蓄電池・ソーラーカーポート）
- Unit B：省エネ・効率化（LED・高効率空調・遮熱・EMS/デマンド監視・断熱）
- Unit C：モビリティ（EV充電器・V2H）
- Unit D：運用・循環（産廃・有価物買取・メンテナンス・パネル処分・ESCOコンサル）

### 8.4 ページテンプレート（2種類）
- **Type A：建物別ページ用（悩み解決型レイアウト / PASONAの法則）**
  - FV → 悩み（共感）→ 解決策（G-NEX）→ 導入事例（証拠）→ CTA
- **Type B：課題別ページ用（カタログ型レイアウト）**
  - FV → 市場背景 → G-NEXの強み → ラインナップ詳細 → FAQ

### 8.5 CMS仕様（フォームフィールド）

**基本情報：**
- ページタイトル
- URLスラッグ
- SEO設定（titleタグ・descriptionタグ）

**ヒーローエリア（FV）：**
- 背景画像：アップロードボタン
- キャッチコピー：テキスト入力
- サブコピー：テキスト入力

**コンテンツエリア（ブロック積み上げ式）：**
- 「悩みリスト」ブロック：箇条書き入力
- 「解決策」ブロック：画像＋テキスト
- 「導入事例」ブロック：Before/After数字入力＋写真

**機能連携：**
- 表示するシミュレーター：チェックボックス選択
  - [ ] 産廃コスト診断（Unit D）
  - [ ] 再エネ診断（Unit A〜C）
- 関連リンク表示の選択

### 8.6 内部リンク構造（SEO）
- 建物別ページから各Unit課題別ページへ相互リンク
- パンくずリスト自動生成

### 8.7 ブログ機能（田邊様追記）
- **【ブログスペース】の設置：** お役立ち情報 ＋ パートナー会社紹介・商材・サービス紹介

## 9. デザイン・UI要件

### 9.1 デザイン方針
- カラー：青系・紺系を基調（#1F3864 / #2E75B6）
- 「ビジネス・信頼感」を演出するデザイン（グリーン・環境系は使用しない）
- グラフ・比較表・カレンダー・写真機能を効果的に活用

### 9.2 確認方法
- Figmaでデモ画面を作成
- クライアントはSlackのスクリーンショット＋テキストで修正指示を送付
- 画面単位で開発を進め、各画面完成後にクライアント確認 → 修正 → 次画面へ

### 9.3 画像素材
- トップ画像（FV）：クライアント提供のJPEGファイルを使用
- 各ページ背景画像：CMSからアップロード

## 10. インフラ・技術要件

### 10.1 ファイルストレージ
- **Amazon S3 連携** による画像・ファイル保存
- S3ライフサイクルポリシー：1年経過後は自動的にGlacier等のアーカイブへ移動

### 10.2 決済
- **Stripe** によるクレジットカード決済（Stripe Checkout）
- リード課金の都度決済に使用

### 10.3 パフォーマンス
- ページ読み込み速度：3秒以内（LCP目標）
- モバイルファーストレスポンシブデザイン
- 画像自動圧縮・リサイズ（スマホ撮影画像を1/10程度に）
- 1枚最大10MB・1投稿合計100MBの制限

### 10.4 セキュリティ
- SSL/TLS（HTTPS）
- 個人情報保護法準拠
- XSS・SQLインジェクション対策
- 定期バックアップ

### 10.5 SEO
- 構造化データ（Schema.org）対応
- サイトマップ自動生成
- PageSpeed Insightsスコア 80以上（目標）
